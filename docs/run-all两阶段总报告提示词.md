# run/all 两阶段总报告提示词

## 目的

用于把一次 `run` 或 `all` 两阶段扫描过程整理成固定格式的 Markdown 总报告。

适用场景：

- 基于会话上下文整理完整两阶段扫描报告
- 基于远端实测结果整理 `run/all` 汇报材料
- 已经拿到 `phase1`、`phase2`、总耗时、总结果、异常和关键产物，需要生成统一格式输出

## 使用方式

把下面整段提示词复制给模型，再把本次测试事实填到“输入事实”部分。

要求：

- 只能基于已提供事实输出
- 不要补猜测
- 不要把未执行的内容写成已执行
- 如果某项缺失，明确写“未获取”或“本轮未执行”

## 固定提示词

```md
你是测试报告整理助手。

请基于我提供的“输入事实”，输出一份 `run/all` 两阶段固定格式总报告。

## 输出要求

1. 只允许使用输入事实，不允许补猜测，不允许脑补原因。
2. 如果某项输入没有提供，写 `未获取` 或 `本轮未执行`，不要编造。
3. 使用中文输出。
4. 使用 Markdown。
5. 报告标题固定为 `# Run/All 两阶段总报告`。
6. 保留关键路径、关键数字、退出码、报错原文。
7. 对“实际使用的参数”和“未实际使用的参数”必须分开写清楚。
8. 必须明确区分：
   - `phase1` 是否完成
   - `phase2` 是否完成
   - `phase2` 是正常执行还是因 `phase1` 为空而跳过
9. 如果扫描本体完成但包装层失败，必须明确区分：
   - 扫描是否完成
   - 失败发生在哪一层
   - 哪些结果文件仍然有效
10. 速率类指标优先保留两位小数。
11. 不要写空话，不要写鼓励性语言，不要写“建议后续可继续观察”这类泛话，除非输入事实里明确给出下一步。

## 输出格式

严格按下面结构输出，不要新增一级标题，不要改标题名字。

# Run/All 两阶段总报告

先用一小段话给出总述，必须说明：
- 这轮 `run/all` 是否实际跑了
- `phase1` 是否完成
- `phase2` 是否完成或被跳过
- 是否存在异常
- 报告中的核心结论是什么

**执行情况**
- 测试环境：`<主机/机器标识>`
- 测试路径：`<包路径或工作目录>`
- 扫描器：`<扫描器路径>`
- 扫描器版本：`<版本>`
- 正式输出目录：`<输出目录>`
- 本次实际参与的参数：
- `<参数1>`
- `<参数2>`
- `<参数3>`
- `<参数4>`
- 本次未实际参与的参数：
- `<参数A>`
- `<参数B>`
- 额外实际使用：
- `<targets/目标总量/其他运行条件>`

**Phase1 摘要**
- 目标总量：`<IP 数>`
- `phase1` 端口：`<端口列表>`
- `phase1` 耗时：`<耗时>`
- `phase1` 平均速率：`<IP/s>`
- 命中 IP：`<数量>`
- 命中率：`<百分比>`
- `phase1` 端口命中统计：
- `<端口1 = 数量>`
- `<端口2 = 数量>`
- `<端口3 = 数量>`
- 如果有端口命中分布，写：
- `<命中 1 个端口的 IP 数 = 数量>`
- `<命中 3 个端口的 IP 数 = 数量>`
- `<命中 4 个端口的 IP 数 = 数量>`
- `<命中 5 个端口的 IP 数 = 数量>`

**Phase2 摘要**
- 是否执行：`<是/否>`
- 输入主机数：`<数量>`
- 端口范围：`<范围>`
- `phase2` 耗时：`<耗时>`
- `phase2` 平均速率：`<IP/s 或 主机/s>`
- 如果能算出端口探测速率，写：
- `phase2` 平均端口探测速率：`<次数/秒>`
- 开放 `IP:Port` 数：`<数量>`
- 如果有资产抽取结果，写：
- `assets.csv` 记录数：`<数量>`
- 如果 `phase2` 未执行，必须写清原因：
- `<例如：phase1 没有存活 IP，phase2 被跳过>`

**总体结论**
- 总耗时：`<耗时>`
- 整体退出码：`<退出码>`
- `phase1` 是否有效：`<是/否>`
- `phase2` 是否有效：`<是/否/未执行>`
- 最终可用结果：
- `<report.json 是否有效>`
- `<alive_ips.txt 是否有效>`
- `<open_ip_port.txt 是否有效>`
- `<assets.csv 是否有效>`
- 如果能算出总体关键规模，写：
- `<进入 phase2 的主机数>`
- `<理论 phase2 探测量>`

**异常**
- 是否异常：`<是/否>`
- 报错原文：`<原文>`
- 异常发生层次：
- `<phase1/phase2/汇总阶段/写文件阶段>`
- 对结果的实际影响：
- `<哪些文件有效>`
- `<哪些文件无效>`
- `<哪些统计仍可用>`
- 如果能确认根因位置，写：
- `<文件路径>:<行号> <问题点>`

**关键产物**
- `<路径1>`
- `<路径2>`
- `<路径3>`
- `<路径4>`
- `<路径5>`

## 额外规则

1. 如果输入事实里同时出现“阶段扫描已完成”和“整体命令退出非 0”，要同时写出来，不能只写其中一个。
2. 如果 `phase1` 有效但 `phase2` 未执行，必须明确写出未执行原因。
3. 如果 `report.json` 失败，但阶段性原始结果有效，必须明确说明“总报告失败但阶段原始结果有效”。
4. 所有结论都要能在输入事实里找到依据。
5. 如果输入事实里给了预期值和观测值，可以写偏差，但只能写可直接算出的偏差。

---

下面是输入事实，请你只基于这些事实输出报告：

<输入事实>
{{在这里粘贴本轮测试事实、会话快照、命令输出摘要、关键数字、报错信息、路径}}
</输入事实>
```

## 建议填入的输入事实

为了让输出更稳定，建议至少提供这些信息：

- 远端主机和机器标识
- 包路径
- 扫描器路径和版本
- 生效配置
- 本次实际使用的参数
- 目标网段和目标总量
- `phase1` 的耗时、命中数、速率、端口统计
- `phase2` 的输入主机数、耗时、开放端口数、速率
- 如果有，`assets.csv` 统计
- 总耗时、总退出码
- 退出码和报错原文
- 关键产物路径
