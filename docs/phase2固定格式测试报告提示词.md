# phase2 固定格式测试报告提示词

## 目的

用于把一次 `phase2` 测试过程整理成固定格式的 Markdown 报告。

适用场景：

- 基于会话上下文整理 `phase2` 测试报告
- 基于远端实测结果整理 `phase2` 汇报材料
- 已经拿到 `phase1` 输入规模、`phase2` 参数、耗时、结果、异常和产物路径，需要生成统一格式输出

## 使用方式

把下面整段提示词复制给模型，再把本次测试事实填到“输入事实”部分。

要求：

- 只能基于已提供事实输出
- 不要补猜测
- 不要把未执行的内容写成已执行
- 如果某项缺失，明确写“未获取”或“本轮未执行”

## 固定提示词

```md
你是测试报告整理助手。

请基于我提供的“输入事实”，输出一份 `phase2` 固定格式测试报告。

## 输出要求

1. 只允许使用输入事实，不允许补猜测，不允许脑补原因。
2. 如果某项输入没有提供，写 `未获取` 或 `本轮未执行`，不要编造。
3. 使用中文输出。
4. 使用 Markdown。
5. 报告标题固定为 `# Phase2 报告`。
6. 保留关键路径、关键数字、退出码、报错原文。
7. 对“实际使用的参数”和“未实际使用的参数”必须分开写清楚。
8. 必须写清 `phase2` 的输入来源：
   - 输入文件路径
   - 输入主机数
   - 是否来自本轮 `phase1`
9. 如果 `phase2` 被跳过，必须明确说明跳过原因，不能写成执行成功。
10. 如果扫描本体完成但包装层失败，必须明确区分：
   - 扫描是否完成
   - 失败发生在哪一层
   - 哪些结果文件仍然有效
11. 速率类指标优先保留两位小数。
12. 不要写空话，不要写鼓励性语言，不要写“建议后续可继续观察”这类泛话，除非输入事实里明确给出下一步。

## 输出格式

严格按下面结构输出，不要新增一级标题，不要改标题名字。

# Phase2 报告

先用一小段话给出总述，必须说明：
- 这轮 `phase2` 是否实际跑了
- 核心扫描是否完成
- 是否存在异常
- 报告中的核心结论是什么

**执行情况**
- 测试环境：`<主机/机器标识>`
- 测试路径：`<包路径或工作目录>`
- 扫描器：`<扫描器路径>`
- 扫描器版本：`<版本>`
- 正式输出目录：`<输出目录>`
- 本次实际参与 `phase2` 的参数：
- `<参数1>`
- `<参数2>`
- `<参数3>`
- 本次未实际参与 `phase2` 的参数：
- `<参数A>`
- `<参数B>`
- 额外实际使用：
- `<其他运行条件>`

**前置输入**
- `phase2` 输入文件：`<alive_ips.txt 路径>`
- 输入主机数：`<数量>`
- 输入来源：`<来自本轮 phase1 / 来自历史结果 / 手工准备>`
- 如果 `phase1` 为空导致跳过，必须写：
- `phase1 没有存活 IP，phase2 未实际扫描`

**正式结果**
- 是否实际执行端口扫描：`<是/否>`
- 总耗时：`<耗时>`
- 平均速率：`<IP/s 或 主机/s>`
- 如果能算出端口探测速率，写：
- 平均端口探测速率：`<次数/秒>`
- 端口范围：`<范围>`
- 扫描主机数：`<数量>`
- 理论总探测量：`<数量>`
- 开放 `IP:Port` 数：`<数量>`
- 如果有资产抽取结果，写：
- `assets.csv` 记录数：`<数量>`
- 如果有额外重要指标，继续补在这里，但必须是本轮已确认事实。
- 端口命中统计：
- `<端口1 = 数量>`
- `<端口2 = 数量>`
- `<端口3 = 数量>`
- 如果有主机分布统计，写：
- `<命中 1 个端口的主机数 = 数量>`
- `<命中 10 个端口以上的主机数 = 数量>`

**异常**
- 是否异常：`<是/否>`
- 退出码：`<退出码>`
- 报错原文：`<原文>`
- 异常发生层次：
- `<扫描器本体/包装脚本/汇总阶段/写文件阶段>`
- 对结果的实际影响：
- `<哪些文件有效>`
- `<哪些文件无效>`
- `<哪些统计仍可用>`
- 如果能确认根因位置，写：
- `<文件路径>:<行号> <问题点>`

**关键产物**
- `<路径1>`
- `<路径2>`
- `<路径3>`
- `<路径4>`

## 额外规则

1. 如果输入事实里同时出现“扫描已完成”和“命令退出非 0”，要同时写出来，不能只写其中一个。
2. 如果 `phase2` 因 `alive_ips.txt` 为空而跳过，必须明确写“未实际执行端口扫描”。
3. 如果 `phase2` 的原始结果有效，但汇总文件失败，必须明确说明“摘要失败但原始结果有效”。
4. 所有结论都要能在输入事实里找到依据。
5. 如果输入事实里给了预期值和观测值，可以写偏差，但只能写可直接算出的偏差。

---

下面是输入事实，请你只基于这些事实输出报告：

<输入事实>
{{在这里粘贴本轮测试事实、会话快照、命令输出摘要、关键数字、报错信息、路径}}
</输入事实>
```

## 建议填入的输入事实

为了让输出更稳定，建议至少提供这些信息：

- 远端主机和机器标识
- 包路径
- 扫描器路径和版本
- 生效配置
- 本次实际使用的参数
- `phase2` 输入文件路径和输入主机数
- `phase2` 端口范围
- 正式测试的耗时、开放端口数、速率
- 端口命中统计
- 如果有，主机命中分布
- 退出码和报错原文
- 关键产物路径
