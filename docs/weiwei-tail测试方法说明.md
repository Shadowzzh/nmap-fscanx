# `weiwei-tail` 测试方法说明

## 1. 目的

本文档说明本次在 `weiwei-tail` 上如何执行扫描测试，以及如果后续不是测试当前的 `1000000` IP 实验环境，而是测试用户直接提供的一个 `IP` 段，我需要哪些前提信息，最终会产出什么结果。

## 2. 本次实际测试对象

本次主要测试的是 `weiwei-tail` 上已经存在的“`1000000` IP 稀疏环境”。

目标范围为：

- `10.240.0.0/13`
- `10.248.0.0/14`
- `10.252.0.0/15`
- `10.254.0.0/16`
- `10.255.0.0/18`
- `10.255.64.0/23`
- `10.255.66.0/26`

总量正好为 `1000000` 个 IP。

已知活 IP 为：

- `10.240.1.10`
- `10.248.2.20`
- `10.255.66.30`

本次统一测试端口为：

- `22`
- `80`
- `443`
- `445`
- `3389`

经实际验证，这 `3` 个活 IP 当前只有 `22` 端口响应，其余 `80/443/445/3389` 不响应。

## 3. 测试前的准备与核实

本次不是直接“拿脚本就跑”，而是先做了下面这些核实：

1. 阅读环境说明文档：
   - `context/weiwei当前100万IP环境核实记录.md`

2. 在 `weiwei-tail` 上确认工具可用：
   - `tmux`
   - `zmap`
   - `masscan`
   - `fscanx`
   - `/usr/bin/time`

3. 在 `weiwei-tail` 上确认实验环境仍存在：
   - `dummy0` 上是否仍挂着活 IP
   - `blackhole` 路由是否仍覆盖上述 `7` 个 CIDR

4. 在 `weiwei-tail` 上确认活 IP 的实际端口响应：
   - 对 `10.240.1.10`
   - `10.248.2.20`
   - `10.255.66.30`
   分别测试 `22/80/443/445/3389`

这样做的原因是先确认：

- 目标环境还在
- 工具真的能跑
- 活主机和端口口径没有变化

否则直接跑基准会把环境问题和工具问题混在一起。

## 4. 本次实际执行方式

### 4.1 `masscan` 与 `fscanx`

这两项测试是在 `weiwei-tail` 内部直接执行的，执行方式参考 `scripts/run_bench.sh` 的口径：

- 目标输入文件放到远端 `/tmp`
- 通过 `tmux` 启动任务，避免 SSH 断开导致中途退出
- 最终把原始日志拉回本地 `out/`

`masscan` 的核心参数是：

- `-iL <targets.txt>`
- `-p22,80,443,445,3389`
- `--rate 100000`
- `--wait 1`

`fscanx` 的核心参数是：

- `-hf <targets.txt>`
- `-p 22,80,443,445,3389`
- `-np`
- `-t 4000`
- `-time 1`
- `-nocolor`
- `-no`

### 4.2 `zmap`

`zmap` 不能像 `masscan` 一样直接对 `22,80,443,445,3389` 做一次多端口扫描，因此当前脚本已经调整为：

- 按 `22/80/443/445/3389` 五个端口分别执行
- 每个端口单独生成 `zmap-port-<port>.txt`
- 再汇总为一个 `zmap.txt`

本次在 `weiwei-tail` 上直接执行 `zmap` 时，还遇到了一个额外问题：

- 默认参数下，`zmap` 会因为网关 MAC 检测失败而直接退出

因此本次补测 `zmap` 时，额外传入了：

- `-G 6c:b1:58:54:09:3b`

这个值来自 `weiwei-tail` 上当前默认网关 `192.168.4.1` 的邻居表记录。

## 6. 当前这轮测试的结论

在 `weiwei-tail` 本机内部直接扫这套 `1000000` IP 稀疏环境时：

- `fscanx` 发现了 `3` 个活 IP 的 `22` 端口
- `masscan` 跑通了，但结果为 `0`
- `zmap` 在补上网关 MAC 后也跑通了，但结果仍为 `0`

这说明：

1. 当前剩下的问题已经不是“脚本跑不起来”
2. 而是 `zmap` / `masscan` 在这种“本机直扫本机 `dummy0 + blackhole`”拓扑下，没有像 `fscanx` 一样打出活 IP

## 7. 如果不是测 `weiwei-tail` 的 `1000000` IP，而是你直接给我一个 IP 段

可以，我能按同样思路进行测试并给出报告。

但我需要你至少明确下面这些信息：

1. 目标范围
   - 一个 `CIDR`
   - 多个 `CIDR`
   - 或一个 `targets.txt`

2. 从哪台机器发起测试
   - `weiwei-tail`
   - 本机
   - 还是其他 SSH 主机

3. 测试端口口径
   - 如果你不指定，我默认还是用：
     - `22,80,443,445,3389`

4. 是否只跑某一个工具，还是三者一起跑
   - `zmap`
   - `masscan`
   - `fscanx`
   - 或三者对比

5. 是否确认对目标范围有授权
   - 如果是你自己的测试网段或明确授权网段，可以继续
   - 如果不是，不能直接扫描

   - 放到 `docs/` 或 `out/` 下

## 9. 需要提前说明的限制

即使给我一个新的 `IP` 段，我也不会默认假设三种工具一定都能“公平比较”。

需要先看实际环境：

- 目标是否可达
- 是否是本机扫本机
- 是否经过隧道
- 是否有 `blackhole` 路由
- 是否需要给 `zmap` 指定 `gateway-mac`

也就是说：

- 我可以测试
- 也可以给报告
- 但会先核实环境，再决定最终采用的具体跑法
