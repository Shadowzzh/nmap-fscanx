# fscanx 字段支持情况说明

## 1. 文档目的

这份说明回答两个不同层面的问题：

1. 上游 `fscanx` 原始 JSON 输出里，可能出现哪些字段类型
2. 当前 `prod-lite` 两阶段流程，最终真正整理进 `report.json` 的字段有哪些

如果只看 `prod-lite` 最终交付结果，结论会比直接看 `fscanx` 原始输出更保守。

## 2. 调研范围与依据

本说明基于以下材料整理：

- 当前仓库中的两阶段脚本 `scripts/fscanx_pipeline.sh`
- 当前发布包入口脚本 `prod-lite/libexec/nmap-fscanx-entry.sh`
- 当前仓库样本输出 `out/nmap-fscanx-vs-nmap-20260424-095405/fscanx/.../normalized.json`
- 上游仓库 `https://github.com/killmonday/fscanx`
- 2026-04-24 拉取的上游提交：`95f791aa99f6939ba516e6c37f890ffc547af282`

其中，上游 `-json` 输出格式的核心依据来自：

- `common/log.go`：定义 JSON 结构只有 `type` 和 `text`
- `common/flag.go`：定义 `-json`、`-nmap` 等参数
- `Plugins/portscan.go`、`Plugins/webScanTask.go`、`Plugins/ms17010.go`：决定 `Port`、`Product`、`OsInfo` 等记录如何生成

## 3. 结论摘要

先看最终结论：

- 当前 `prod-lite` 最终明确支持的字段只有 `IP` 和 `开放端口`
- `是否在线` 不能按标准主机存活字段来宣称支持
- `OS` 在上游原始输出中属于部分支持，但当前 pipeline 没有写入最终报告
- `应用服务` 与 `Web 产品指纹` 在上游原始输出中属于部分支持，但当前 pipeline 没有写入最终报告
- `应用服务版本` 只能算弱支持，少量目标会在 `Product` 文本里带出类似 `iis:10.0` 这样的线索，但当前没有结构化落地

一句话概括：

- `fscanx` 原始输出比 `report.json` 丰富
- `prod-lite` 当前交付口径远窄于上游原始能力

## 4. 上游 `fscanx` 的 JSON 输出到底是什么结构

上游 `fscanx` 的 `-json` 并不是“为每类资产设计好的结构化对象集合”，而是把原本日志行包装成：

```json
{
  "type": "Port",
  "text": "open\t192.168.1.162:22\t\n"
}
```

也就是说：

- 最外层稳定字段只有 `type`
- 具体资产信息绝大多数仍然塞在 `text` 里
- 后续是否能提取出 `IP`、`端口`、`标题`、`证书`、`版本`，取决于你有没有再做二次解析

这也是为什么当前 `prod-lite` 只稳定提取了 `Port` 里的 `IP` 与 `IP:PORT`。

## 5. 当前可见的原始记录类型

结合上游源码和当前仓库样本，当前至少能稳定见到这些 `type`：

| `type` | 来源 | 含义 | 是否结构化 |
| --- | --- | --- | --- |
| `msg` | 通用日志 | 任务开始、阶段提示、普通信息 | 否 |
| `Port` | 端口探测 | 开放端口，文本里通常包含 `IP:PORT`，有时包含协议名 | 部分结构化 |
| `Product` | Web/指纹探测 | URL、状态码、标题、Web 指纹、证书、跳转来源等 | 否，主体仍是文本 |
| `OsInfo` | SMB/MS17010/RDP 等插件 | 操作系统及部分主机名/工作组信息 | 否，主体仍是文本 |
| `alive` | 扫描结束摘要 | 当前命中的端口汇总 | 否 |
| `scan` | 扫描结束摘要 | 耗时等结束信息 | 否 |

注意：

- `type` 是稳定的
- `text` 的内容格式不是强 schema，属于“有规律的日志文本”

## 6. 字段支持矩阵

下面区分“上游原始输出是否能拿到”和“当前 `prod-lite` 最终报告是否交付”。

| 字段 | 上游原始输出 | 当前 `prod-lite` 最终输出 | 结论 |
| --- | --- | --- | --- |
| `IP` | 支持 | 支持 | 当前第一阶段输出 `alive_ips.txt`，第二阶段输出 `open_ip_port.txt` |
| `是否在线` | 部分支持 | 不支持 | 当前流程不是标准活性发现流程，不能把结果等价成 `is_online` |
| `开放端口` | 支持 | 支持 | 当前最终报告的核心结果就是唯一 `IP:PORT` 列表 |
| `OS` | 部分支持 | 不支持 | 原始输出可见 `OsInfo`，但当前未写入 `report.json` |
| `应用服务` | 部分支持 | 不支持 | 原始输出可见 `Product` 以及少量协议线索，但当前未结构化输出 |
| `应用服务版本` | 弱支持 | 不支持 | 少量文本里可见版本，如 `iis:10.0`，但没有统一结构化字段 |
| `Web 标题` | 部分支持 | 不支持 | 在 `Product` 文本里可见页面标题 |
| `证书信息` | 部分支持 | 不支持 | 在 `Product` 文本里可见 `Cert:` 片段 |

## 7. 为什么当前最终报告只剩 `IP` 和 `开放端口`

当前两阶段脚本的提取逻辑非常明确：

- 第一阶段：只从 `type == "Port"` 的记录中提取唯一 `IP`
- 第二阶段：只从 `type == "Port"` 的记录中提取唯一 `IP:PORT`

对应脚本：

- `scripts/fscanx_pipeline.sh` 的 `extract_alive_ips`
- `scripts/fscanx_pipeline.sh` 的 `extract_open_ip_ports`
- `write_phase1_summary`
- `write_phase2_summary`
- `write_report`

这意味着：

- `Product`
- `OsInfo`
- `alive`
- `scan`

这些类型当前都不会进入最终 `report.json`。

## 8. 为什么“是否在线”不能算当前支持

这是最容易被误读的点。

当前第一阶段执行的是：

- 扫描固定重点端口，默认 `22,80,443,445,3389`
- 命中这些端口任意一个，才进入第二阶段

因此当前 `phase1/alive_ips.txt` 的真实语义是：

- “命中第一阶段重点端口的 IP 列表”

而不是：

- “标准意义下的在线主机列表”

下面几种主机都会被当前第一阶段漏掉：

- 在线，但没开放这 5 个端口
- 在线，但这些端口临时被防火墙拦截
- 在线，但服务延迟较大，超时参数过小

所以：

- 当前不能把 `alive_ips.txt` 当成严格的 `is_online=true` 清单
- `alive` 类型记录也不能当作主机在线字段

## 9. `alive` 字段到底是什么

样本中的 `alive` 记录长这样：

```json
{"type":"alive","text":"ports(5): 443,445,80,22,3389"}
```

它表示的是：

- 本轮扫描命中过哪些端口

它不是：

- 每个 IP 的存活状态
- 每个网段的存活主机数
- 标准主机发现结果

因此，`alive` 只能看成摘要信息，不能看成资产字段。

## 10. `Product` 字段能提供什么

`Product` 记录主要来自 Web 访问和指纹识别逻辑，样本里常见内容包括：

- URL
- HTTP 状态码
- 页面标题
- Web 指纹
- 证书信息
- 跳转来源
- 部分产品名或版本线索

例如样本里能看到：

- `[nginx]`
- `[harbor]`
- `[openresty-product]`
- `[vmware esxi]`
- `[iis:10.0]`

但要注意三件事：

1. 这些信息仍然主要存在于 `text` 文本中
2. 同一个 `Product` 里可能混合标题、证书、跳转来源、多个指纹
3. 当前 `prod-lite` 没有把它拆成结构化字段

所以它更适合叫：

- “原始证据”

而不是：

- “已经稳定整理好的资产字段”

## 11. `OsInfo` 字段能提供什么

`OsInfo` 主要来自 SMB / MS17010 / RDP 相关探测结果。

样本里可见内容包括：

- `Windows 10 Enterprise 19045`
- `Windows 11, version:10.0.26100`
- `Windows 10.0.19041 [WORKGROUP]`

这说明：

- 某些 Windows 目标确实能被拿到 OS 线索
- 但覆盖面明显不是所有主机
- 它不是统一资产建模后的 `os.name`、`os.version`、`hostname` 等字段

因此它只能算：

- 原始输出部分支持

不能算：

- 当前交付结果已支持

## 12. `-nmap` 与当前 `prod-lite` 的关系

上游 `fscanx` 的 `-nmap` 用来开启协议识别能力。

它的主要影响是：

- `Port` 记录第三列更可能出现协议名，而不是空
- 某些端口会按识别出的协议调用更合适的后续插件
- 服务识别能力会比纯 TCP connect 更强

但是当前 `prod-lite` 的两阶段脚本没有传 `-nmap`。

这带来两个直接后果：

1. 当前 `Port` 记录基本只能稳定提取 `IP:PORT`
2. 当前最终报告更不应该宣称支持“协议名”“服务名”“服务版本”

换句话说：

- 上游有协议识别能力
- 当前 `prod-lite` 默认流程没有启用这部分能力

## 13. 最终可以怎么表述字段支持

如果是对外或对内做口径统一，建议直接写成下面这样：

### 13.1 当前 `prod-lite` 最终交付口径

- 明确支持：`IP`、`开放端口`
- 不应直接宣称支持：标准 `是否在线`
- 原始输出中部分可见，但当前最终报告未落地：`OS`、`应用服务`、`Web 指纹`、`证书信息`
- 只能算弱支持，且未结构化：`应用服务版本`

### 13.2 如果只讨论上游 `fscanx` 原始 JSON

- 能看到多种 `type`
- `Port`、`Product`、`OsInfo` 都能携带资产线索
- 但大部分信息仍在 `text` 中，属于“日志型 JSON”，不是标准资产对象

## 14. 如果后续要补强字段，有两条路线

### 方案 A：继续深挖 `fscanx` 原始输出

可以在当前 `normalized.json` 基础上新增解析，尝试产出：

- `os`
- `hostname`
- `service_name`
- `service_version`
- `web_title`
- `web_fingerprints`
- `cert_subject`
- `evidence`

优点：

- 基于现有流程改动小
- 不增加二次扫描成本

限制：

- 原始格式本质上仍是日志文本
- 解析规则会越来越重
- 版本字段很难做到统一、稳定

### 方案 B：把 `fscanx` 作为快扫入口，再补二次识别

推荐流程：

1. `fscanx` 做快速筛选
2. 对命中的 `IP:PORT` 再跑 `nmap -sV`
3. 对需要 OS 的目标再跑 `nmap -O` 或专项探测

优点：

- 字段更标准
- `OS / Service / Version` 准确性通常更高

限制：

- 扫描成本更高
- 流程更复杂

## 15. 最终结论

如果问题是“当前 `prod-lite` 最终交付支持哪些字段”，答案非常明确：

- 明确支持：`IP`、`开放端口`
- 不应直接宣称支持：`是否在线`
- 原始输出里部分支持，但当前未落地：`OS`、`应用服务`
- 只能算弱支持，且当前未落地：`应用服务版本`

如果问题是“上游 `fscanx` 原始 JSON 有没有更多线索”，答案也是肯定的：

- 有，但主要以 `type + text` 的日志型 JSON 形式存在
- 当前 `prod-lite` 还没有把这些线索整理成最终资产字段
