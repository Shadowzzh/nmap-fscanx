# `ZMap`、`masscan`、`fscanx` 内网活跃主机发现速度对比记录

## 1. 测试时间

- 测试日期：`2026-04-23`
- 测试机器：当前本机 `Mac`
- 测试方式：
  - 使用本地 `tmux` 会话串行执行
  - `ZMap` 和 `masscan` 通过 `sudo` 提权执行
  - 每个工具单独输出原始日志
  - 最后汇总为统一结果表

## 2. 测试目的

本轮测试用于对比下面 `3` 个工具在“小范围内网 + 常见高价值 TCP 端口”场景下的活跃主机发现速度：

- `ZMap`
- `masscan`
- `fscanx`

这里的“活跃主机”不是 `ICMP ping` 意义上的“活”，而是：

- 目标 IP 在 `22,80,443,445,3389` 这 `5` 个 TCP 端口中，只要任意端口有响应，即视为“活跃”

## 3. 测试范围与口径

### 3.1 目标范围

本轮实际测试网段为：

- `192.168.1.0/24`
- `192.168.20.0/24`
- `192.168.4.0/24`

合计可用主机地址总数为：

- `762`

### 3.2 统一端口口径

三种工具统一测试端口为：

- `22`
- `80`
- `443`
- `445`
- `3389`

说明：

- 本轮不是“全端口扫描”
- 本轮也不是“纯 IP 保活测试”
- 更准确的定义是“基于 `5` 个常见 TCP 端口响应的活跃主机发现测试”

### 3.3 实际执行口径

本轮最终采用的有效口径如下：

- `ZMap`
  - 使用 `allowlist` 方式输入目标 CIDR
  - 最终有效写法为 `-w /tmp/scan-bench/targets.cidrs`
- `masscan`
  - 使用 `-iL /tmp/scan-bench/targets.txt`
- `fscanx`
  - 使用 `-hf /tmp/scan-bench/targets.txt`
  - 端口参数为 `-p 22,80,443,445,3389`

补充说明：

- 中途曾用 `ZMap` 的 `-I` 方式测试小规模目标列表，但在当前场景下明显偏慢，已中断并废弃
- `ZMap` 帮助信息本身也提示：当目标少于 `1000000` 个 IP 时，优先使用 `allowlist`，不要优先使用 `-I`

## 5. 结果汇总

| 工具 | 耗时（秒） | 退出码 | 结果行数 | 唯一主机数 | 说明 |
| --- | ---: | ---: | ---: | ---: | --- |

- `ZMap` 约为 `masscan` 的 `2.8` 倍速度
- `ZMap` 约为 `fscanx` 的 `8.2` 倍速度
- `masscan` 约为 `fscanx` 的 `2.9` 倍速度

## 6. 结果解读

### 6.1 关于速度

如果只看本轮“小范围内网 + `5` 个端口”的纯耗时：

- `ZMap` 最快
- `masscan` 第二
- `fscanx` 最慢

### 6.2 关于结果数量

虽然 `ZMap` 和 `masscan` 的结果行数都为 `34`，但二者命中的主机集合并不完全一致。

可确认的信息：

- `ZMap` 命中唯一主机数：`25`
- `masscan` 命中唯一主机数：`28`
- 两者交集主机数：`14`

这说明：

- 在小范围、多端口、单次实测下，`ZMap` 与 `masscan` 的发现集合并不会天然完全一致
- 不能把任一工具的一次结果直接当作最终完整真相

### 6.3 关于 `fscanx`

`fscanx` 的结果行数明显更多，不代表它“发包更快”，而是因为它除了端口探测，还顺手做了下面这些事情：

- 协议识别
- Web 指纹识别
- `RDP` 信息识别
- `SMB` 结果输出

因此：

- `fscanx` 不适合和 `ZMap`、`masscan` 做完全等口径的“纯高速发包工具”比较
- 但它很适合作为第二阶段补充识别工具

## 7. 本轮可直接得出的结论

### 7.1 如果只追求速度

在本轮测试口径下，优先级为：

- `ZMap`
- `masscan`
- `fscanx`

### 7.2 如果兼顾速度和工程通用性

在常见内网资产发现实践里：

- 第一阶段可优先用 `ZMap` 或 `masscan` 做高速筛活
- 第二阶段再用 `fscanx` 做协议识别和资产细化

### 7.3 关于适用范围

这份测试记录只适用于下面这个测试口径：

- 小范围内网
- `3` 个 `/24`
- 共 `762` 个 IP
- `5` 个常见 TCP 端口

这份结果不能直接外推为：

- 全端口扫描结论
- `ICMP` 保活结论
- `ARP` 保活结论
- `1000000` IP 规模下的最终性能结论

## 8. 后续建议

如果后续要进一步用于“百万 IP”场景决策，建议补做下面两类测试：

- 全端口测试：
  - 重点看 `masscan` 与 `fscanx` 的差异
- 百万级目标测试：
  - 重点看 `ZMap` 与 `masscan` 在大规模目标上的真实吞吐差异

如果目标是工程落地，建议采用两阶段方案：

- 第一阶段：`ZMap` 或 `masscan` 高速筛活
- 第二阶段：`fscanx` 做识别、指纹和补充资产信息
